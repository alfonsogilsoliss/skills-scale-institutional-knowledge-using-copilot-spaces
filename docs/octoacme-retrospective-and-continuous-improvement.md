# OctoAcme — Retrospectiva y Mejora Continua

## Propósito
Capturar aprendizajes y convertirlos en mejoras accionables.

## Cuándo
Después de cada sprint, release o hito importante. También después de incidentes.

## Participantes
Se recomienda que todos los roles del equipo participen en la retrospectiva: PM, PdM, Desarrolladores, Líder Técnico, Diseñador UX/UI, QA y, cuando sea relevante, el Stakeholder Externo para incorporar perspectivas externas.

## Estructura
- ¿Qué salió bien?
- ¿Qué podría mejorar?
- Elementos de acción (responsable, fecha límite)
- Seguimiento a elementos de acción previos

## Cómo Facilitar una Retrospectiva
- Limitar el tiempo: 45–75 minutos según el tamaño del equipo
- Usar un tablero de ideas anónimas si es necesario para fomentar la sinceridad
- Priorizar 2–3 elementos de acción principales para evitar sobrecarga
- Asignar responsables claros para cada elemento de acción (evitar solapamientos o vacíos de accountability)

## Seguimiento de Mejoras
- Agregar elementos de acción al backlog del proyecto o como issues con responsables y fechas claras
- Revisar acciones pendientes en la sincronización semanal del PM
- El Líder Técnico lidera las mejoras relacionadas con deuda técnica o estándares de desarrollo
- QA lidera las mejoras relacionadas con calidad, cobertura de pruebas y gestión de defectos

## Checklist de Retrospectiva
- [ ] Todos los roles del equipo convocados (PM, PdM, Desarrolladores, Líder Técnico, Diseñador UX/UI, QA)
- [ ] Stakeholder Externo invitado cuando haya temas de validación o satisfacción relevantes
- [ ] Seguimiento a elementos de acción previos realizado
- [ ] Elementos de acción nuevos documentados con responsables y fechas
- [ ] Resumen de la retro compartido con el equipo

## Cultura de Mejora Continua
- Medir el impacto de los elementos de acción
- Celebrar las mejoras y realizar cambios pequeños e iterativos
- Asegurar que las lecciones aprendidas se reflejen en los procesos y checklists del equipo
